# 垂直切片技术设计

状态：**已批准实施**<br>
适用版本：Vertical Slice 0.1<br>
引擎：Godot 4.6.3-stable、GDScript<br>
最后更新：2026-07-13

> 技术设计服务于 `VERTICAL_SLICE.md`，无权扩大产品范围或改写叙事事实。

稳定 ID、方法、信号和数据字段以 `IMPLEMENTATION_CONTRACTS.md` 为准；本文只定义系统职责和协作结构。

## 1. 技术目标

- 用最少系统稳定跑通双轮体验，而不是提前建设完整版框架。
- 所有关键状态可观察、可重置、可从开发菜单直接进入。
- 基础住宅结构只有一个来源，轮回变化不复制整张地图。
- 文本、音频和变化配置与控制代码分离，便于叙事迭代。
- 切片中不引入无法证明必要性的插件、持久化层或通用事件编辑器。

## 2. 版本与项目约束

- 项目、编辑器和导出模板统一使用 `Godot 4.6.3-stable`。
- 开发期间锁定该版本；升级前必须建立新的决策记录并完整回归两个结尾。
- 使用标准 GDScript，不启用 C# / Mono。
- 目标渲染分辨率为 640×360，像素资产使用整数缩放；窗口允许 2×、3×、全屏显示。
- 目标帧率 60 FPS。所有叙事时间线使用与帧率无关的计时。
- 首发导出目标仅 Windows x86_64。

## 4. 场景结构

```text
Main
├── House
│   ├── BaseGeometry
│   ├── Rooms
│   ├── Props
│   ├── InteractionPoints
│   ├── MutationLoop1
│   └── MutationLoop2
├── Player
├── EventDirector
├── AudioDirector
├── PostProcess
├── UI
└── DevTools                  # 仅开发构建启用
```

### House

- `BaseGeometry` 是墙体、门洞和碰撞的唯一来源。
- 房间通过稳定的 `room_id` 标记：`bedroom`、`hallway`、`kitchen`、`child_room`、`living_room`。
- 可变物品拥有稳定的 `object_id`，不得依赖运行时自动生成的节点名保存叙事状态。
- `MutationLoop1/2` 只负责可见性、替换资源、文本变体和小范围属性变化，不复制碰撞地图。

### EventDirector

负责阶段迁移和跨系统编排：冻结/恢复输入、播放惩罚时间线、切换变化集、淡入淡出、进入结尾。它不保存长期数据，也不直接包含叙事文本。

### AudioDirector

管理环境、界面、事件和心跳四类总线。峰值限制在统一混音规则内；“主音量”必须影响全部总线。

## 5. 状态模型

`GameState` 是唯一的运行进度权威来源。建议状态如下：

```text
phase:
  TITLE
  LOOP_1
  PUNISHMENT_1
  LOOP_2
  ENDING_FACE
  ENDING_AVOID

cycle_index: 0 | 1
completed_fragments: { kitchen_receipt, child_drawing, wedding_photo }
first_room_loop_1: empty | kitchen | child_room | living_room
behavior_echo_played: bool
clock_attempts: int
clock_solved: bool
final_memory_opened: bool
ending_committed: bool
```

### 状态不变量

- `PUNISHMENT_1` 只能在三个碎片均完成时进入，且每次进程最多进入一次。
- `cycle_index == 1` 时必须处于 `LOOP_2` 或某个结尾阶段。
- `first_room_loop_1` 一旦写入不得覆盖。
- `clock_solved` 只能在第二轮为真。
- `final_memory_opened` 为真之前不能提交任何结尾。
- `ending_committed` 为真后禁止恢复玩家移动或写入另一个结尾。
- 场景重载不得清空 `GameState`；返回标题或开始新游戏必须显式重置全部进度。

设置状态与进度状态分离。音量、全屏和降低闪烁可以写入本地设置文件；剧情进度不得写入磁盘。

## 6. 状态迁移

```text
TITLE
  └─ new_game → LOOP_1
       └─ all_fragments_completed → PUNISHMENT_1
            └─ timeline_finished → LOOP_2
                 ├─ clock_solved → final_memory_opened
                 │    ├─ hold_memory_confirmed → ENDING_FACE
                 │    └─ hold_sleep_confirmed → ENDING_AVOID
                 └─ remain_in_loop → LOOP_2

ENDING_FACE / ENDING_AVOID
  └─ ending_finished → TITLE + reset_progress
```

任何未列出的迁移都视为缺陷。开发工具可以调用受控跳转，但必须先建立该阶段所需的合法前置状态。

## 7. 交互系统

### 接口

所有可调查对象实现同一交互约定：

- 返回稳定 `interaction_id`；
- 报告当前是否可交互；
- 提供提示文本键；
- 接收一次明确的 `interact()` 调用；
- 在文本完成前保持忙碌，防止重复触发。

玩家交互探测使用短距离朝向区域。候选排序必须固定：是否位于正前方、距离、`interaction_id`。禁止每帧依赖节点遍历顺序，否则重叠物品会随机切换。

### 碎片提交

碎片脚本不得在第一次按键时立即完成。流程为：

1. 锁定当前交互；
2. 播放文本与短暂记忆；
3. 文本系统发出完成信号；
4. `GameState` 幂等写入碎片 ID；
5. 更新音频层；
6. 若总数变为三，下一空闲帧请求惩罚阶段。

幂等意味着重复调用不会重复计数、重复播放完成效果或再次启动惩罚。

## 8. 文本与本地化

- 场景和脚本只引用文本键，不包含最终中文句子。
- 中文基线来源是 `NARRATIVE_BEATS.md`，实际运行数据放在 `data/dialogue/`。
- 每个键使用稳定语义名称，例如 `fragment.kitchen.first.01`，禁止使用整句中文作为键。
- 文本框必须支持自动换行、标点停顿、立即显示当前页和继续下一页。
- 字体必须覆盖所有使用字符，并在 `ASSET_CREDITS.md` 登记嵌入与再分发许可。
- 不依赖文字颜色单独表达选择或危险；同时提供形状、图标或明确措辞。

## 9. 轮回变化系统

垂直切片只需要两个命名变化集：`loop_1_default` 和 `loop_2_response`。

每条变化记录：

- 目标 `object_id`；
- 变化类型：显示、隐藏、纹理替换、位置微调、文本组切换、音频状态；
- 目标值；
- 可选前置条件；
- 用于调试的设计意图说明。

应用变化集必须可重复调用且结果一致。第二轮开始时按以下顺序执行：冻结输入、恢复基础状态、应用第二轮固定变化、应用行为回应配置、把玩家放到卧室出生点、淡入、恢复输入。

切片不允许随机选择变化。所有变化都必须能从设计文档追溯到叙事目的。

## 10. 房间行为记录

房间入口使用不重叠的触发区域。只在 `LOOP_1` 且 `first_room_loop_1` 为空时记录厨房、儿童房或客厅；走廊和卧室永不成为该值。

第二轮进入匹配房间时：

1. 检查 `behavior_echo_played == false`；
2. 读取房间对应文本键；
3. 播放一次字幕；
4. 成功排入文本队列后将标志写为真。

若第一轮因开发跳转没有记录房间，第二轮不播放回应，不得使用虚构的默认行为句。

## 11. 惩罚时间线

使用单一可检查的时间线资源编排画面、灯光、音频和移动倍率。禁止实现不可见敌人节点或碰撞伤害。

要求：

- 进入时关闭普通调查交互，但保留移动和出口门专用文本；
- 暂停游戏时所有时间线、音频渐变和移动倍率同步暂停；
- “降低闪烁”不改变总时长，只替换高对比变化曲线；
- 无论玩家站在哪个房间，最终黑屏和第二轮重置都能完成；
- 事件结束前再次收到触发请求必须被忽略并记录开发警告。

## 12. 时钟知识门槛

- 第二轮才启用调整界面。
- 时与分分开调整，分钟以一分为步进；允许按住方向键连续变化。
- 目标为 02:17，比较标准化后的 24 小时数值。
- 错误确认增加 `clock_attempts`，不关闭界面、不重置已输入时间。
- 第三次错误后切换收据复读文本，玩家下一次查看收据时得到加强提示。
- 正确确认设置 `clock_solved`，播放暗格动画；该流程幂等。

为避免机械输入拖慢节奏，界面可以提供十分钟快速步进，但不得直接显示目标时间。

## 13. 结尾提交

“面对”和“回避”都使用三秒按住确认组件：

- 组件必须明确显示动作和松开可取消；
- 暂停时按住进度暂停；
- 进度完成瞬间先原子写入结尾状态，再锁定输入；
- 两个交互共享同一个提交守卫，避免同帧触发双结尾；
- 结束后返回标题并调用进度重置；设置保留。

## 14. 音频与画面安全

- 音频总线：Master、Ambience、Event、UI、Heartbeat。
- 导入前统一检查响度和峰值，禁止通过极端音量制造惊吓。
- 空间音频只用于可定位的住宅声音；执行者的最终接近声故意使用非空间或贴近监听者的混音。
- 全屏亮度变化不得超过 `VERTICAL_SLICE.md` 的闪烁上限。
- “降低闪烁”默认关闭，但首次内容提示直接提供设置入口。

## 15. 调试与可观察性

开发构建提供：

- 新游戏状态重置；
- 标记任意碎片完成；
- 以合法前置状态进入第一轮惩罚；
- 跳到第二轮并选择第一房间记录；
- 设置时钟错误次数和解锁状态；
- 跳到两个结尾；
- 在屏幕角落显示当前阶段、轮回、碎片、第一房间和关键布尔值。

发布构建必须移除或禁用这些入口。关键非法迁移在开发构建中记录清晰错误，发布构建中安全忽略，不能导致崩溃或软锁。

## 16. 测试策略

- 垂直切片以场景级自动检查、状态模型测试和人工体验测试结合。
- 纯状态逻辑应与节点表现分离，使六种碎片顺序、幂等提交、合法迁移和重置可以快速测试。
- 视觉、音频、碰撞、文本节奏和情感效果必须人工验证。
- 不要求为了切片引入大型测试框架；如果采用插件，必须说明维护成本并锁定版本。
- 完整验收矩阵见 `PLAYTEST.md`。

## 17. 实施顺序

1. 项目初始化、精确引擎版本记录、输入和设置。
2. 单一基础住宅、玩家移动、碰撞和房间识别。
3. 统一交互与外置文本系统。
4. `GameState`、三个碎片和任意顺序完成。
5. 惩罚时间线与第二轮重置。
6. 固定变化、行为回应和时钟门槛。
7. 两个结尾、标题回流和完整状态重置。
8. 调试工具、音画安全选项和发布导出。
9. 功能验收、盲测、修订。

每一步结束时都必须保持主分支可运行；禁止在基础探索未完成前建设完整版知识图谱或程序化生成器。
