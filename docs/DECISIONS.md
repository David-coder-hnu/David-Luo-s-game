本文件记录会显著影响范围、叙事或架构的决定。编号一经使用不得复用。修改既有决定时不得直接删除历史结论，应追加“已取代”状态和新的决策编号。

## 已接受决定

### D-001：采用俯视角 2D 像素探索

- 状态：接受
- 理由：最适合五房间住宅的观察、回访和空间记忆，也与目标受众熟悉的交互语言一致。
- 后果：垂直切片期间不评估横版方案；地图、碰撞与镜头按俯视角设计。

### D-002：使用 Godot 4.6.3-stable 与 GDScript

- 状态：接受
- 理由：2D、音频、场景和跨平台导出能力满足项目需求，且许可适合公开仓库。4.6.3 是已有补丁维护的稳定版本，避免在切片阶段追随刚发布且仍在修复回归问题的更新分支。
- 后果：项目和导出模板统一使用 `4.6.3-stable`；开发周期内不得无评估升级引擎版本。
- 版本依据：[Godot 官方发布归档](https://godotengine.org/download/archive/)。

### D-003：先制作可结束的双轮垂直切片

- 状态：接受
- 理由：单轮加重置只能验证功能，第二轮才能验证环境回应和“害怕继续”的核心体验。
- 后果：切片目标时长 12–15 分钟，完成两轮后返回标题；不制作无限可玩内容。

### D-004：故事永恒循环，作品本身有结尾

- 状态：接受
- 理由：保留“没有逃脱”的主题，同时尊重玩家时间并提供情感收束。
- 后果：结尾可以改变主角面对真相的姿态，不能取消犯罪事实或轮回。

### D-005：主角身份与完整罪行延迟揭示

- 状态：接受
- 理由：玩家应先把房子理解为谜，再逐步意识到自己与惨剧的关系。
- 后果：公开 README、商店文案和预告片不得泄露主角杀害妻女；垂直切片也不揭示完整犯罪经过。

### D-006：采用剧情旗标与玩家推理的混合知识系统

- 状态：接受
- 理由：旗标保证叙事一致，真正的推理动作兑现“知识存在于玩家脑中”的承诺。
- 后果：每个主要章节至少有一个无任务标记的知识门槛；自动解锁只能作为节奏辅助。

### D-007：惩罚段落采用脚本化压迫，不制作追逐 AI

- 状态：接受
- 理由：不可见实体的简单轴向追踪会被碰撞阻挡，也无法稳定控制时长和节奏。
- 后果：玩家在段落中仍可移动，但灯光、门、音频和最终黑屏由事件时间线控制；结局不可通过走位逃脱。

### D-008：垂直切片不跨启动保存

- 状态：接受
- 理由：切片时长短，持久化不能帮助验证核心体验，反而增加退出时状态一致性成本。
- 后果：场景切换和轮回内保留知识；关闭进程后重新开始。所有文档和测试统一采用此规则。

### D-009：房屋使用单一基础场景与变化层

- 状态：接受
- 理由：复制 A/B 完整场景容易产生碰撞、脚本和内容漂移。
- 后果：基础结构保持稳定，轮回变化通过独立节点组或数据资源启停。

### D-010：环境变化优先手工编排

- 状态：接受
- 理由：心理恐怖需要可解释的作者意图；随机性不是差异化本身。
- 后果：垂直切片和完整版前半段均使用命名变化集；只有在手工循环验证成功后才评估受约束的程序化选择。

### D-011：家庭暴力采用间接表现，并维护受害者主体性

- 状态：接受
- 理由：直接猎奇展示会削弱心理恐怖，也可能把妻女降格为主角的惩罚工具。
- 后果：使用生活物品、文字、空间和声音呈现；酒精不作为免责；内容提示在公开页面可见。

### D-012：首个版本以简体中文和 Windows 为目标

- 状态：接受
- 理由：限制测试矩阵，优先验证体验。
- 后果：所有文本必须外置，禁止写死在场景脚本中；后续语言和平台不属于切片完成条件。

### D-013：代码与文档采用 MIT，资产逐项登记许可

- 状态：接受
- 理由：公开代码不应让第三方素材的许可状态含糊。
- 后果：每个美术、音频和字体资产必须登记作者、来源、许可和修改情况；未登记资产不得发布。

### D-014：锁定 640×360、16 像素网格和克制低饱和色板

- 状态：**已由 D-018 部分取代**
- 理由：统一网格、色板和整数缩放能让 AI 辅助资产仍保持同一作品感，并保证中文 UI 的可读空间。
- 后果：像素资产、相机、UI 与后处理遵循 `ART_BIBLE.md`；高分辨率生成图只作概念参考和品牌展示。

### D-015：五房间采用十字型中央枢纽布局

- 状态：接受
- 理由：厨房、儿童房和客厅需要拥有近似公平的首选距离，第二轮行为回应才不是关卡构图偏置的结果。
- 后果：墙体、门洞、关键物和碰撞以 `LEVEL_BLOCKOUT.md` 坐标为基线；第二轮不得改变基础通路。

### D-016：跨模块接口先于并行实现锁定

- 状态：接受
- 理由：多个 AI 并行施工时，稳定 ID、方法、信号和字段不统一会造成高额返工。
- 后果：实现必须遵循 `IMPLEMENTATION_CONTRACTS.md`；需要变更契约时先更新决策和所有消费者，不得在单个任务中私自改名。

### D-017：生成式图像仅用于叙事特写与品牌概念图

- 状态：接受
- 理由：复杂纸张、照片与录音带特写能从生成式构图和材质中受益，但世界图集、角色和 UI 需要严格网格与可重复编辑。
- 后果：世界瓦片、碰撞道具、角色和 UI 使用确定性 SVG/像素源；生成特写必须保存母版与提示，并通过最近邻缩放、有限色无抖动量化和人工对照后才能进入运行时目录。

### D-018：运行时视觉升级为 32 像素瓦片并引入实机证据门

- 状态：接受
- 理由：16×16 资产在 640×360 画布中无法同时提供可信住宅材质、稳定调查物轮廓和接近品牌主视觉的氛围；静态尺寸与色板检查也不能替代美术验收。
- 后果：逻辑地图仍使用 50×38 的瓦片坐标和既有门洞/交互布局；环境视觉单元改为 32×32，秦峥改为 32×48，UI 画布保持 640×360。八个核心语义色继续锁定，允许使用 `ART_BIBLE.md` 登记的八个材料过渡色。任何运行时资产在拥有 Godot 第一轮、第二轮与惩罚状态截图前最高只能标记为 `REVIEW`。
