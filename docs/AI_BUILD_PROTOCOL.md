# AI 高速构建协议

状态：**实施流程已锁定**<br>
适用范围：垂直切片开发<br>
最后更新：2026-07-13

## 1. 目的

本协议把制作文件转化为 AI 可以执行、人工可以验收的小任务。目标不是让 AI 自由“做一个恐怖游戏”，而是让它在边界清楚、输入完整、结果可测试的情况下高速完成工程工作。

任何任务如果仍要求执行者自行决定核心剧情、视觉语言、房间结构或状态规则，就尚未达到可实施状态。

## 2. 权威输入

AI 开始任务前只按 `docs/README.md` 的权威顺序读取与任务相关的文件。最低输入集合：

- 所有任务：`DECISIONS.md`、`VERTICAL_SLICE.md`；
- 剧情/文本：加 `NARRATIVE_BEATS.md`；
- 场景/状态：加 `TECHNICAL_DESIGN.md`；
- 跨模块接口：加 `IMPLEMENTATION_CONTRACTS.md`；
- 美术：加 `ART_BIBLE.md` 和 `ART_REVIEW.md`；
- 地图：加 `LEVEL_BLOCKOUT.md`；
- UI：加 `UI_UX_SPEC.md`；
- 音频：加 `AUDIO_DIRECTION.md`；
- 任何资产生产：加 `ASSET_MANIFEST.md` 和 `ASSET_CREDITS.md`；
- 验收：对应 `PLAYTEST.md` 条目。

历史归档不得作为提示输入，除非任务明确要求比较历史决定。

## 3. Definition of Ready

一个任务只有满足以下条件才可以交给 AI：

- 玩家可感知结果用一句话说清；
- 明确允许修改和禁止修改的文件范围；
- 所有依赖任务已经完成；
- 输入资产存在或指定合法占位方式；
- 状态前置、成功结果和失败行为明确；
- 至少一个可执行或可观察的验收方法；
- 不包含“适当”“有氛围”“看起来不错”等无量化要求，除非同时指向美术圣经的具体规则；
- 不需要执行者做新的产品决策。

未满足时先修文档或拆任务，不让 AI 猜。

## 4. 标准任务包

每次任务使用以下结构：

```text
Task ID:
Outcome:
Why it matters:

Required reading:
Dependencies:
Allowed files:
Forbidden changes:

Inputs / assets:
State preconditions:
Expected behavior:
Failure behavior:
Accessibility / narrative constraints:

Acceptance checks:
Manual verification:
Evidence to return:
```

AI 返回内容必须包含：修改文件、关键设计映射、执行的检查、未解决风险。不得只回答“已完成”。

## 5. 任务粒度

- 单任务理想修改 1–6 个紧密相关文件。
- 单任务应能在一次上下文内实现并验证。
- 不把“建立完整游戏框架”“实现所有 UI”“完成美术”作为一个任务。
- 同一任务不同时处理状态架构、叙事改写和视觉重制。
- 机械批量工作可以扩大文件数，但必须有稳定模板和回滚方式。

## 7. 并行规则

允许并行：

- F-02 与文本数据准备；
- P-02 资产修订与 N 系列灰盒逻辑；正式导入和截图仍等待 G-01/G-02；
- P-04 音频素材准备与 N-03 时间线接口；
- 文档校对与不接触相同文件的实现任务。

禁止并行：

- 两个任务同时修改 `GameState`；
- 文本键结构未锁定时同时批量写剧情与 UI；
- 基础地图未验收时制作最终家具碰撞；
- 惩罚时间线接口未锁定时做最终混音自动化。

共享文件必须有单一所有者，其他任务通过接口或数据文件接入。

## 8. AI 行为边界

AI 不得自行：

- 新增房间、角色、结局、谜题或跨会话系统；
- 把占位文字改成新的剧情事实；
- 使用历史归档覆盖当前决定；
- 下载或生成未登记许可的发布资产；
- 为了测试方便削弱内容安全或退出功能；
- 把概念主视觉宣称为实机截图；
- 在任务范围外“大规模顺手重构”。

发现规格冲突时停止相关实现，指出冲突位置和最小决策，不选择自己喜欢的答案。

## 9. 评审顺序

每个实现任务按以下顺序评审：

1. **范围**：是否只做任务包内容；
2. **事实**：是否符合决策和叙事；
3. **状态**：是否保持不变量和幂等；
4. **体验**：反馈、节奏和视觉是否符合专项规格；
5. **安全**：退出、闪烁、音量、内容责任表达；
6. **证据**：测试、截图、录屏、日志；
7. **维护**：命名、许可、文档同步。

前一层失败时不讨论后续层的精细打磨。

## 10. Definition of Done

任务完成必须同时满足：

- 所有验收检查通过；
- 没有新增 P0/P1；
- 自动检查或人工步骤可被他人复现；
- 玩家可见结果有截图或短录屏；
- 状态变更有调试证据；
- 新文本、资产和设置均进入对应权威文件或登记表；
- 未使用的占位、调试日志和死代码已删除；
- 返回结果明确说明未验证的部分，不能把“未测试”写成“应该可用”。

## 11. 变更控制

实现中若发现设计需要改变：

1. 记录观察证据；
2. 指出受影响的决定编号；
3. 提出最小替代方案与代价；
4. 更新 `DECISIONS.md` 并追加取代关系；
5. 同步专项规格和测试；
6. 再创建新的实现任务。

禁止先修改代码再倒推文档合理化。
