## Problem Statement

设计一款 2D 像素心理恐怖游戏——一个男人酗酒后杀害了妻子和孩子，悔恨中放火烧屋自焚，堕入地狱。在地狱中，他不断在自己曾经的房子里轮回：失忆、探索、找回记忆碎片、面对地狱之神、被折磨至死、再次失忆——永恒的惩罚循环。

核心设计挑战：做一款"进度本身就是陷阱"的游戏——玩家越想理解故事，离痛苦就越近。

## What Makes This Cool

这个设计颠覆了游戏的基础语法。所有游戏都训练玩家相信"进度是好的、知识是力量、胜利是目标"。在这里，三条全是谎言：

- **恢复记忆 ≠ 胜利，恢复记忆 = 惩罚的前奏。** 每次"我想起来了"和"他要来了"是同一瞬间。
- **玩家越接近真相，离地狱之神的酷刑就越近。**
- **没有打破轮回的隐藏结局。** 循环是永恒的。玩家能做的只有"比上一轮多了解一点"。

这不是靠 jump scare 吓人的游戏。它靠的是让玩家在面对真相时，自己想要退缩——但又无法停止探索。

参照系：Ib、魔女之家、狂父的 2D 像素传统 + Outer Wilds 的知识驱动探索 + The Stanley Parable 的哲学颠覆。

## Constraints

- **目标受众：** 单人沉浸体验——关灯、戴耳机、独自玩
- **风格：** 2D 像素探索（俯视角 tilemap 起步），日式 RPG Maker 恐怖传统
- **操作方式：** 键盘 WASD/方向键移动，E 键或鼠标点击交互物品，原型阶段不支持手柄
- **进步方式：** 纯知识驱动——玩家不获得新能力，改变的只有玩家脑子里的理解
- **环境变化（v2）：** 程序化微妙变化是 v2 目标——物品偏移、色调变暗、照片细节模糊。**原型阶段通过手摆场景变体实现**（做两套略有不同的房子布局，重置时切换），不做运行时程序化生成
- **开发目标：** 周末原型 → 验证核心循环 → 迭代扩展

## Premises

1. **知识是唯一的进步** — 记忆碎片以日记、闪回、物品描述形式累积。新区域和新事件由"玩家知道了什么"解锁，而不是"角色获得了什么能力"。
2. **潜意识级的环境变化** — 房子每轮微妙变化，不被主动宣布。理想情况下，玩家会在第三四轮时自己意识到"等等……这里之前不是这样的"。
3. **没有真正的胜利** — 找回全部记忆 → 地狱之神出现 → 折磨至死 → 重新失忆 → 新的轮回。循环是永恒的。
4. **2D 像素探索 + 沉浸式音效** — 参照日式像素恐怖传统。音效设计需要达到头戴耳机级别的沉浸感——环境音、空间音频、策略性沉默。
5. **"进度"本身是一个陷阱** — 越接近真相，离地狱之神的酷刑就越近。每一次"我记起来了"都同时是"他要来了"的信号。

## Cross-Model Perspective

独立 Claude 子代理冷读评审的核心发现：

**最酷的未考虑版本：** 玩家的真实行为变成游戏机制——跨会话追踪玩家行为，房子回应玩家本人。详见下方「Future Ideas (v2+)」章节。

**用户最兴奋的核心：** "进度是陷阱"——这不是一个恐怖游戏点子，这是一个游戏设计论纲。让他们兴奋的不是吓玩家，而是颠覆电子游戏的基础语法。

**引擎建议：** Godot 4.x — MIT 协议、完整 2D 管线、空间音频支持、跨平台导出。免费获得渲染/输入/碰撞/场景管理，需要自己构建记忆碎片系统、轮回管理器和程序化变异规则。

**周末原型建议：** 一个房子（4-5 房间）、一个完整轮回、一次重置、三个记忆碎片。先证明"进度是陷阱"的核心情感节拍，再建系统。

## Approaches Considered

### Approach A: Godot 周末原型 (最小可行) — **已选择**

用 Godot 4.x 先做一个 10 分钟可玩原型——一个房子、一个完整轮回、一次重置。手摆所有内容，零程序化生成。验证核心循环的情感冲击力，再决定是否扩展。

- 工作量: S — 一个周末出可玩原型
- 风险: 低 — 每一步都有可玩的东西
- 优点: 最快验证核心假设；每个里程碑可玩可测；如果核心循环不成立，损失只有几天
- 缺点: 原型代码可能无法直接复用；程序化轮回在 v1 中不存在

### Approach B: RPG Maker MZ 叙事优先

用 RPG Maker MZ 内置事件系统快速做完整叙事，不碰程序化。

- 工作量: S — 开发速度是纯代码的 3-5 倍
- 风险: 中 — 程序化能力极有限，后续可能需要整体迁移
- 优点: 最快路径到完整叙事体验；天然日式像素恐怖血统
- 缺点: 程序化变化实现困难；迁移时内容无法直接复用

### Approach C: Godot 架构先行 (理想架构)

从 Day 1 搭建完整系统架构——知识图谱、轮回管理器、地狱之神接近度系统、叙事脚本层。

- 工作量: L — 系统搭建需要数周
- 风险: 中 — 可能在系统上花几周后发现核心循环不成立
- 优点: 程序化轮回一等公民；长期维护友好
- 缺点: 漫长的系统开发期后才有可玩内容；作为学习项目容易失去动力

## Recommended Approach

**Approach A: Godot 周末原型。**

先证明核心循环成立，再投资系统。以下是原型的具体技术规格：

### 两天里程碑

**Day 1：可走动的房子 + 记忆碎片**
- Godot tilemap 搭建 4-5 个房间（卧室、走廊、厨房、儿童房、客厅）
- 键盘移动 + 碰撞检测
- 2 个可交互记忆碎片（一个明显物品如血迹、一个需要观察的物品如儿童画）
- 全局知识状态字典（`Global.knowledge = {}`）
- 物品交互 → 文本框弹出 → 屏幕闪烁 → 字典写入条目

**Day 2：地狱之神事件 + 轮回重置**
- 第 3 个记忆碎片（隐藏——结婚照，检查时发现妻子脸部被刮花）
- 收集全部 3 个碎片后触发地狱之神事件
- **触发效果（简化版）：** 第三个碎片被收集时，画面饱和度在 3 秒内降至 30%，低频心跳声渐入。（红色边缘噪点作为 Day 2 的可选扩展——先确保饱和度 + 音频跑通，再加噪点着色器。）
- **追逐机制：** 神是**不可见的**——没有精灵、没有碰撞体可见轮廓。神通过一个不可见的 Area2D 节点在场景中移动，玩家通过心跳声的音量判断距离（越近越响）。神沿四个方向之一直线移动（朝向玩家的 X 或 Y 方向），速度比玩家慢 20%。因为房子空间封闭且走廊狭窄，死亡在 15-20 秒内是**必然的**——这不是一个可以"赢"的追逐。接触 = 立刻死亡。
- **追逐无需 A* 寻路：** 使用简单的方向追踪（`if god.x < player.x: move_right`），在 tilemap 碰撞体限制下自然形成追逐感。Day 2 不引入寻路系统。
- 死亡后无菜单——屏幕 2 秒黑屏，直接淡入回到卧室
- 场景切换至手动制作的变体：血迹消失、儿童画内容改变、照片面朝下

**第 3 轮及之后（原型限制）：** 原型只有 2 套场景变体（A → B → A → B……）。这是原型的已知限制——2 轮之后模式会被玩家识别。真正的"无尽变化"需要 v2 的程序化变异系统。

### 具体的知识解锁示例

示例：玩家在儿童房检查了一本日记 → `Global.knowledge["child_feared_basement"] = true` → 原本锁着的地下室门现在可以打开。实现方式：地下室门的 `_ready()` 检查该标志，若为 true 则显示"可以打开"，若为 false 则显示"锁住了"。

### 存档与轮回

**游戏内轮回（死亡 → 重置）：** 自动发生，无菜单。死亡后 2 秒黑屏 → 淡入回到卧室。知识状态保留在内存中（`Global.knowledge` 不清空），场景切换到变体。
**新游戏会话（退出 → 重新打开）：** 原型不支持跨会话存档。退出游戏 = 进度丢失，下次启动从头开始。这是原型的有意限制——每次"会话"就是一次完整的游戏体验。跨会话持久化（包括知识状态跨启动保留和玩家行为追踪）属于 v2。
**追逐中退出：** 如果玩家在追逐中按 Alt+F4，下次启动从卧室重新开始，知识状态恢复到本轮碎片收集前的状态。

### 边界情况

- 玩家未收集全部碎片时可以无限探索——没有时间限制。但没有任何碎片进展时无事发生，促使玩家主动寻找。
- 如果玩家在追逐前退出游戏：下次启动从卧室重新开始，知识状态保留，碎片进度保留。

如果原型核心循环不成立（氛围不到位、情感冲击不够），任何程序化系统和架构优化都是浪费。如果成立，Approach C 的架构以原型为基础逐步搭建。

## Open Questions

1. **视角选择：** 俯视角（类似 Ib/Mad Father 的探索感）还是横版（更强的空间压迫感）？原型阶段用**俯视角 tilemap**。两种视角对应完全不同的关卡设计和碰撞系统，一旦选定切换成本高——建议原型完成后根据测试感受做最终决定，而非中途切换。
2. **记忆碎片的具体实现：** ~~日记本？记忆墙？闪回过场？~~ **已锁定：** 原型使用**可交互物品 → 文本框弹出 → 屏幕短暂闪烁 → 写入全局知识状态字典**。这是最小可行的实现，后续可替换为更丰富的呈现方式。
3. **每轮时长：** 参考 Lily's Well 的约 15 分钟/轮。原型阶段一轮 10 分钟内，确保循环节奏紧凑。
4. **地狱之神的表现形式：** ~~像素角色？阴影？~~ **已锁定：** 原型使用**纯音频 + 屏幕后处理效果**——最少的视觉呈现往往最恐怖。具体参数见 Recommended Approach。

## Success Criteria

原型阶段"完成"的技术定义：
- [ ] 玩家可以在房子中自由探索（4-5 个房间），WASD 移动 + E 键交互
- [ ] 3 个记忆碎片可被发现——一个明显、一个需要观察、一个隐藏
- [ ] 全局知识状态字典正确记录已发现的碎片
- [ ] 全部碎片收集后触发地狱之神事件（屏幕效果 + 音频 + 追逐 + 死亡）
- [ ] 重置后场景有可感知的变化（至少 3 处不同）
- [ ] 完整的"醒来 → 探索 → 发现 → 死亡 → 重置"循环可以跑通

> **注意：** "关灯戴耳机感到真实的压迫感"是情感目标而非技术输出——原型验证的是核心循环的结构和节奏，不是已打磨的惊吓效果。氛围打磨是迭代过程，不要期待第一版原型就吓到自己。

## Future Ideas (v2+)

以下特性在跨模型评审中被提出，值得记录但**不在原型范围内**：

- **跨会话玩家行为追踪：** 游戏静默记录玩家的行为模式（犹豫时间、避开哪些房间、游戏时间），并在后续轮回中用这些数据回应玩家本人。"你每次都先检查厨房。43 次了。你为什么一直回来？"——这是整个项目的最终极形态，但需要独立的追踪系统和大量叙事设计工作。
- **运行时程序化变异：** 替代原型中的手摆场景变体，实现真正的程序化物品偏移、色调变化、布局微调。
- **多层轮回叙事：** 超过 2 轮的完整叙事弧线，每轮揭示新的真相层次。

## What I noticed about how you think

- 你在描述游戏剧情时几乎没有犹豫——整段故事一气呵成。这意味着这个故事你已经想了很久，不是一时兴起的点子。
- 你的审美选择始终在同一个方向收敛：心理恐怖而非生存恐怖、微妙变化而非显著变化、日式像素传统而非 3D 写实。这不是随机选择——你有一个清晰的审美体系。
- 你选了"纯知识驱动"作为进步方式。这比 Metroidvania 式的技能解锁要难做得多——没有能力升级作为奖励钩子，全靠叙事本身。选这个说明你信任你的故事足够强。

## Failure Modes

| 路径 | 失败方式 | 检测 |
|------|---------|------|
| 知识状态字典 | 键名拼写错误导致门永远打不开 | 手动测试——发现碎片后检查门是否可交互 |
| 场景变体切换 | 重置后物品状态未正确更新 | 手动测试——死亡后观察 3 处变化是否生效 |
| 过场追逐 | 着色器未正确触发或音频未播放 | 手动测试——第三个碎片后观察画面+听音频 |
| 全局知识跨场景 | `Global.knowledge` 在场景重载后丢失 | 场景重载后检查已发现碎片是否仍记录 |
| 边界情况：碎片重复交互 | 同一碎片多次触发导致字典重复写入 | 添加 `if not knowledge.has(key)` 守卫 |
